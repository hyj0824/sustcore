# 构建系统

该系统的构建系统是 theflysong 结合 linux 的构建系统, 从头开始设计并实现的. 该构建系统的最初版本可见于 theflysong 早期的作品 [TayhuangOS](https://github.com/TayhuangOS-Development-Team/TayHuangOS/tree/refactor), 后面 theflysong 在该系统的基础上进行了大量的重构和优化, 使其支持多平台构建 + 制作 initrd 并链接到内核中. hyj0824 则在此基础上进行了进一步的优化, 使其支持 build/release 模式. 我最终添加了 config.json 配置文件及其自动化生成工具, 使得一方面可以生成 loggers.h 配置头文件, 另一方面可以自动化生成 config.mk 文件并添加架构相关编译选项, 使其可以方便地编译到多个平台上, 并由 hyj0824 最终支持了符合比赛要求的 make all 目标.

Sustcore 的构建系统负责配置生成、静态库构建、模块构建、initrd 整理、内核链接以及 QEMU 运行.
顶层 `Makefile` 负责组织整个流程, 而实际的编译、链接与组件规则则主要放置在在 `script/` 目录下的一系列通用模板中.
同时, 大多数重复逻辑被统一抽象到 component.mk 中, 以便在不同组件中复用.

## 多语言支持

目前项目中主要涉及 C、C++ 与汇编三类源码. 构建系统通过统一的组件模板自动识别并处理这些源文件. 对于大部分组件而言, 它们只需要声明自己的源文件集合, 构建系统就会根据文件类型自动选择对应的编译方式.

Sustcore 中, 不同语言的分工主要体现在:
1. C++ 主要承担内核主体逻辑、能力系统、内存管理、VFS、调度器以及绝大多数模块逻辑.
2. C 主要用于少量更贴近底层 ABI 或与现有运行时接口更契合的部分(string.c 等纯 c 接口).
3. 汇编则主要出现在架构相关的启动、上下文切换、trap 入口、crt 及少量用户态/子系统入口桥接代码中.

此外, 本项目中的 module 与普通宿主机程序不同, 它们需要链接项目自己的运行时支持库并遵循内核规定的入口方式与 ABI,
同时, linux subsystem 则要链接特殊的 linuxss-libc 运行时库以支持 linux 子系统的运行.

## 多平台支持

Sustcore 当前最主要支持的两个平台是 `riscv64` 与 `loongarch64`.
构建系统对多平台的支持最主要通过交叉编译器前缀来实现, 并有额外的特殊处理.
首先, 平台会影响配置文件的读取结果. `config.json` 中每个架构都有自己的一组配置, 包括编译器前缀、附加编译选项、feature 开关、logger 配置以及 QEMU 运行参数.

平台的不同会影响源码的组织方式. 内核与部分模块都同时包含平台无关代码和平台相关代码. 平台无关部分负责描述统一的抽象与公共逻辑, 平台相关部分则负责处理中断入口、上下文布局、启动方式、链接脚本、特定设备初始化等问题. 
构建系统需要在保持整体统一的前提下, 为不同平台挑选不同的源文件、不同的启动对象以及不同的链接方式. 这部分, 通过选择 `riscv64-sources` 与 `loongarch64-sources` 添加源文件来实现

## 配置生成

Sustcore 中, 构建系统通过将 config.json 编译为 config.mk, loggers.h 等若干中间产物, 由 Make 去消费这些结果. 这样做的好处在于:

1. Make 层不需要承担复杂的结构化配置解析逻辑.
2. logger、feature、QEMU 参数与编译选项都可以通过统一配置源控制.
3. 用户可以通过命令行临时覆盖部分设置, 同时仍然保留默认配置文件作为主要入口.

从开发体验上说, 这使得 Sustcore 的构建系统既保留了 Make 的直接性, 又避免了把大量平台配置、日志配置和 feature 开关硬编码在 Makefile 里.

## 组件化构建

Sustcore 的库、模块与内核并不是各自维护完全独立的构建逻辑, 而是都套在同一个组件化框架下. 每个组件主要描述以下几类信息:

1. 它属于静态库、模块还是内核本体.
2. 它由哪些源文件组成.
3. 它依赖哪些库或链接脚本.
4. 在不同平台或不同变体下是否需要调整输出方式.

这种组织方式的好处是: 当项目规模继续扩大时, 构建系统不会退化成一个巨大且难以维护的顶层 `Makefile`. 相反, 顶层只负责"调度", 子组件负责具体信息提供, 两者职责是相对清晰的.
也可以据此快速把握一个目录的定位: `options.mk` 更接近"组件配置", `include.mk` 更接近"源码清单", 而通用构建模板才是真正定义行为的地方.

## initrd 与内核镜像

Sustcore 的构建系统将内核视作一个特殊的模块, 它的构建方式与普通模块类似, 但在链接时会额外把 initrd 作为一个二进制 blob 链接到内核镜像中.
这部分是由 kernel 自身的 `options.mk` 描述的.
对于这个项目而言, module、linux subsystem、init 以及若干测试模块本身是系统可运行性的一部分.
因此, 构建系统会在构建这些模块之后, 进一步把它们整理进 initrd, 再把 initrd 作为附件, 使用 objcopy 转化成中间文件后链接到内核镜像的特定段内.

## build / run / dbg 的含义

对使用者来说, 最常见的几个命令是 `make build`、`make run`、`make dbg` 与 `make all`.

1. `build` 关注的是产物生成, 它要完成库、模块、initrd 与内核镜像的全流程构建.
2. `run` 在构建完成后直接以默认参数启动 QEMU.
3. `dbg` 在构建完成后会让 QEMU 停在最开始的状态, 方便 gdb 远程连接.
4. `all` 则负责比赛要求的产物生成.
