# Atomic 与 Spinlock

本文说明 Sustcore 中两类基础并发原语的职责边界与使用约定:

- `std::atomic<T>`
- `SpinLocker` / `GuardedLock`

它们都用于处理并发访问, 但保护对象的粒度和语义完全不同。选择错误的原语, 往往会导致:

- 数据竞争
- 状态不一致
- 死锁
- 在中断上下文中执行非法操作
- 难以复现的时序错误

## 相关实现

- `include/std/c++/atomic`
- `include/std/c++/bits/atomic_base.h`
- `kernel/spinlock.h`
- `kernel/arch/riscv64/spinlock.h`
- `kernel/driver/int/base.h` 中的 `InterruptGuard`

## 基本原则

先判断你要保护的到底是什么:

1. **单个共享标量**
2. **多个字段组成的复合状态**
3. **容器、链表、映射表**
4. **当前 hart 与中断上下文之间的共享状态**
5. **多个 hart 之间的共享状态**

推荐规则:

- 单个共享标量: 优先使用 `std::atomic<T>`
- 复合状态或共享容器: 使用自旋锁
- 只关闭当前 hart 的中断: 使用 `InterruptGuard`
- 同时涉及当前 hart 中断和其他 hart 并发: 需要明确的锁协议, 不能只靠 `InterruptGuard`

## `std::atomic<T>`

`atomic` 适合表达**独立的共享标量**。所谓“独立”, 是指这个对象本身的读写就能表达完整语义, 不需要同时更新其它结构。

典型例子:

- 计数器
- 标志位
- 简单状态枚举
- 已发布的指针

### 适合的场景

以下场景通常适合 `atomic`:

- 唯一 ID 分配器
- 引用计数
- “是否已完成 / 是否已取消 / 是否需要重调度”这类单字段状态
- 只需要原子读取或原子加减的位图

### 不适合的场景

以下对象不应只靠 `atomic` 保护:

- `std::vector`
- `std::unordered_map`
- 侵入式链表
- 需要同时更新多个字段才能保持语义正确的状态机

例如:

- “从等待队列摘下线程, 再把它改成 READY”
- “向 mount 表插入记录, 同时增加打开文件计数”
- “修改 capability 槽位, 同时调整引用关系”

这类操作的关键问题不是“单次读写要原子”, 而是“多个动作必须作为一个整体完成”。

### 使用建议

- 对计数器优先使用 `fetch_add` / `fetch_sub`。
- 对状态转换优先使用 compare-exchange, 避免裸读改写。
- 不要把 `atomic` 当成无锁容器的替代品。
- 不要把“结构一致性问题”拆成若干原子字段后假装已经线程安全。

### 对齐要求

`atomic` 对对象地址有严格对齐要求。

使用前必须确认:

- 对象本身的布局满足对齐
- 分配器返回的地址满足对齐
- placement new 的目标缓冲区满足对齐
- 对象池没有破坏对齐

如果底层分配器不满足 C++ `operator new` 的基本对齐要求, 那么把普通整数改成 `atomic` 后可能会直接触发未对齐访问异常。

## `SpinLocker`

`SpinLocker` 是最基础的短临界区互斥原语。其语义是:

- 如果锁空闲, 立即获取
- 如果锁被占用, 原地忙等
- 解锁后其它执行流才可进入临界区

它适合保护**复合共享状态**, 尤其是:

- 一个对象的多个字段
- 容器与其中元素的关系
- “状态位 + 队列成员关系”这类必须同时维护的结构

### 接口

```cpp
SpinLocker lock;

lock.lock();
// critical section
lock.unlock();
```

通常更推荐使用 `GuardedLock`:

```cpp
GuardedLock guard(lock);
// critical section
```

`GuardedLock` 采用 RAII 方式自动释放锁, 更适合带错误返回和早退出的代码路径。

### 适合的场景

以下对象通常更适合自旋锁:

- 调度队列
- 等待队列
- 中断域映射表
- VFS 挂载表
- capability 槽位空间
- “位图 + 等待者数组” 这种混合对象

### 不适合的场景

以下情况通常不应使用自旋锁:

- 临界区很长
- 临界区内部可能阻塞或睡眠
- 临界区内部会触发复杂的外部调用且锁顺序不清晰
- 能用单个原子变量表达的问题

## `GuardedLock`

`GuardedLock` 是 `SpinLocker` 的 RAII 包装。

推荐使用方式:

```cpp
Result<void> update_state()
{
    GuardedLock lock(_lock);
    // 修改共享状态
    void_return();
}
```

优点:

- 作用域结束自动解锁
- 错误传播路径更安全
- 避免手工 `unlock()` 漏掉

如果必须跨作用域控制持锁时机, 才考虑直接调用 `lock()` / `unlock()`。

## `InterruptGuard`

`InterruptGuard` 的职责不是互斥所有 CPU, 而是:

- 关闭**当前 hart** 的中断
- 防止当前执行流被本地中断处理程序打断
- 离开作用域后恢复之前的中断状态

它只解决:

- 当前 hart 普通执行路径
- 与当前 hart 中断处理路径

之间的竞争。

它**不能**解决:

- 另一个 hart 同时访问同一共享对象
- 多个 CPU 并发修改同一容器

因此, `InterruptGuard` 不是自旋锁的替代品。

## 选择规则

### 规则一: 单字段用 `atomic`

如果共享语义完全落在一个字段上, 优先 `atomic`。

例如:

- 唯一递增 ID
- 引用计数
- “是否完成” 标志

### 规则二: 容器和多字段状态用自旋锁

只要出现以下任意一种情况, 就应优先考虑自旋锁:

- 修改 `vector` / `map`
- 修改链表
- 同时更新多个字段
- 一个字段变化后必须立刻同步调整另一结构

### 规则三: 中断竞争单独考虑

如果共享对象会被中断处理函数访问:

- 只用普通自旋锁未必足够
- 只用 `InterruptGuard` 也未必足够

必须明确:

- 是否只存在单 hart 竞争
- 是否存在跨 hart 竞争
- 锁是否需要带关中断语义

## 编码约束

### 使用 `atomic` 时

- 先确认对象满足对齐要求
- 优先让共享语义收束在单字段
- 不要把多个相关字段拆成原子后伪装成“线程安全对象”
- 状态机转换优先 CAS, 少用裸读改写

### 使用自旋锁时

- 临界区尽量短
- 持锁期间不要睡眠
- 持锁期间不要做可能阻塞的资源获取
- 如果需要同时持有两把锁, 必须先定义固定锁顺序
- 尽量把“查找 + 修改 + 回滚”都收进同一临界区

### 使用 `InterruptGuard` 时

- 只在确实需要防本地中断重入时使用
- 不要把它当成跨 hart 同步原语
- 与自旋锁组合使用时, 先明确锁协议, 避免隐式依赖

## 简短判断表

| 场景 | 推荐原语 |
| --- | --- |
| 单个计数器 / 标志位 | `std::atomic<T>` |
| 共享链表 / 映射 / 队列 | `SpinLocker` |
| 当前 hart 与本地中断共享状态 | `InterruptGuard` 或带中断语义的锁 |
| 多个字段必须整体一致 | `SpinLocker` |
| 跨 hart 容器修改 | `SpinLocker` |

## 一句话准则

- **单个共享标量用 `atomic`**
- **复合共享状态用自旋锁**
- **`InterruptGuard` 只负责本地中断, 不负责跨 hart 互斥**
